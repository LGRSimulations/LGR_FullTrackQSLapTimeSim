# DeepSeek Lesson Chat Design

## Goal

Add a globally accessible AI chat panel to the simulator workbench. The panel uses the lesson markdown files as its source of truth and routes questions through DeepSeek via a two-step agentic retrieval chain. The system is self-contained, adds under 5MB to the distribution ZIP, and requires no local ML model.

## Architecture

Four components:

1. **Build script** — generates the section index from lesson files once, committed to the repo
2. **Encrypted key file** — distributor encrypts the DeepSeek API key before packaging the ZIP
3. **Chat service** — FastAPI service that runs the two-call retrieval chain
4. **Frontend popover** — toggle button in the tab bar, floating panel that persists across tabs

---

## Component 1: Section Index

**File:** `docs/lessons/index.json`
**Generated by:** `tools/build_lesson_index.py`

The build script walks `docs/lessons/*.md`, splits each file at `##` headings, and calls DeepSeek once per section to produce a one-sentence summary. The result is saved as a JSON array.

Each entry:

```json
{
  "file": "Tyre-Model.md",
  "section": "Where base_mu fits",
  "summary": "base_mu is a global grip scale that raises or lowers the tyre force ceiling across all slip conditions.",
  "heading_level": 2
}
```

The build script is run manually by the distributor and the output is committed. Re-run it whenever lessons are added or significantly changed. The index is read-only at runtime — no index writes during chat.

---

## Component 2: API Key Encryption

**Key file:** `config/deepseek_key.enc`
**Encryption script:** `tools/encrypt_api_key.py`

Encryption uses Fernet symmetric encryption from the `cryptography` package. The Fernet secret is a fixed 32-byte key baked into the Python source at `src/app/services/chat_service.py` as a module-level constant. This is intentional obfuscation for ZIP distribution, not production-grade security.

To produce a key file:

```
python tools/encrypt_api_key.py sk-your-deepseek-key
```

Output: `config/deepseek_key.enc`

To rotate the key, run the script with the new key and replace the `.enc` file. Recipients paste the new `.enc` file into `config/` and relaunch.

The decryption happens once at service startup and the plain key is held in memory for the session.

## Component 3: Chat Service

**File:** `src/app/services/chat_service.py`
**Endpoint:** `POST /api/chat`

### Request schema

```python
class ChatRequest(BaseModel):
    question: str
    history: list[dict]  # [{ "role": "user"|"assistant", "content": str }]
```

### Response schema

```python
class ChatSource(BaseModel):
    file: str     # e.g. "Tyre-Model.md"
    section: str  # e.g. "Where base_mu fits"

class ChatResponse(BaseModel):
    answer: str
    sources: list[ChatSource]
```

### Retrieval chain

**Call 1 — section selection**

System prompt instructs DeepSeek to act as a retrieval filter. It receives the full `index.json` as a formatted list of section names and summaries, plus the user question. It returns a JSON array of section names that are relevant.

Example system prompt fragment:

```
You are a retrieval filter for a Formula Student simulator knowledge base.
Given the user question and the list of available sections below, return a JSON array
of section names that contain information needed to answer the question.
Return only the JSON array, nothing else.

Sections:
- "Tyre Model Intro > One minute mental model": The tyre model translates slip and normal load into contact patch force.
- "Tyre Model Intro > Where base_mu fits": base_mu is a global grip scale ...
...
```

**Call 2 — answer generation**

The selected section names are used to look up the full markdown content from disk. The raw text of each matched section is assembled into a context block. DeepSeek receives:

- System prompt: role as simulator tutor, instruction to ground answers in the provided notes, permission to reason beyond them when the notes do not cover the question
- Context block: raw markdown of retrieved sections
- Conversation history: full `history` list from the request
- User question

The response is returned as `answer`. The section names passed into call 2 are returned as `sources`.

### Error handling

- If `config/deepseek_key.enc` is missing or decryption fails, the endpoint returns HTTP 503 with message "Chat is not configured."
- If `docs/lessons/index.json` is missing, same 503.
- If call 1 returns malformed JSON (not a valid array), fall back to returning all sections (graceful degradation).
- If DeepSeek is unreachable, return HTTP 502 with the raw error message.

## Component 4: Frontend Popover

### Tab bar button

A button labelled `Ask` is appended to the right end of the `.tab-bar`. It does not behave like a tab (no `data-tab` attribute, no section switching). It toggles the chat panel open and closed.

When the panel is open, the button reads `Ask` with a filled/active visual state (green background). When closed, it returns to the inactive tab style.

### Panel

Fixed position, right edge of the viewport, full height below the tab bar. Width 380px. The panel slides in from the right using a CSS transform transition. It sits above page content via `z-index`.

Layout (top to bottom):

- Panel label: `Lesson Assistant`
- Scrollable message thread
- Source pills row (shown below each assistant reply)
- Input bar with a text field and Send button

The panel does not close when the user switches tabs. It closes only when the user clicks the `Ask` button again.

### Message thread

User messages right-aligned, assistant messages left-aligned. Each assistant message has a `sources` row below it showing the cited section names as small clickable pills. Clicking a pill closes the chat panel, switches to the Lesson Guide tab, and loads the lesson file that contains that section. If no sections were retrieved, the sources row is hidden.

The `sources` list returned by the backend includes the filename (e.g. `Tyre-Model.md`) so the frontend can match it against the `.lesson-item[data-src]` attributes already in the sidebar to trigger the correct lesson load.

### Conversation history

Stored as a JS array in memory. Sent in full with every request. Resets on page reload. No persistence between sessions.

### Input behaviour

Send button is disabled while a request is in flight. The input field is also disabled during that period. An inline `...` indicator appears in the thread while waiting.

## Out of Scope

- Streaming responses (DeepSeek streaming API)
- Persistent chat history across page reloads
- Per-lesson contextual mode (chat always has access to all lessons)
- Authentication or per-user key management
